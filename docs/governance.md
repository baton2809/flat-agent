# Governance: FlatAgent

## Risk Register

| Риск | Вероятность / Влияние | Детект | Защита | Остаточный риск |
|---|---|---|---|---|
| Prompt injection через сообщение пользователя | Средняя / Высокое | Логирование всех входящих сообщений; мониторинг аномально длинных промптов | Системный промпт задаёт роль и границы; user input передаётся отдельно от system instructions; GigaChat не имеет доступа к БД напрямую | Низкий - LLM не может выполнять системные команды |
| Prompt injection через результаты поиска (DuckDuckGo) | Средняя / Высокое | Логирование raw search results | Результаты поиска передаются как данные, а не как instructions; LLM получает их в отдельном поле контекста | Средний - зависит от качества разделения ролей в промпте |
| Утечка данных пользователя из SQLite | Низкая / Высокое | Мониторинг файловой системы; alerts при нестандартном доступе к .db файлу | БД хранится только на сервере; нет API endpoint для прямого доступа к БД; backup только внутри VPS | Низкий при правильной настройке сервера |
| Недоступность GigaChat API | Высокая / Высокое | Healthcheck endpoint /api/v1/health проверяет доступность GigaChat | Retry с экспоненциальным backoff (3 попытки); graceful error message пользователю; timeout 30 сек | Средний - нет fallback LLM в PoC |
| Недоступность ЦБ РФ API | Средняя / Среднее | Логирование HTTP-ошибок при обращении к cbr.ru | Кэш последнего успешного значения ставки; пометка в ответе о давности данных | Низкий - ставка меняется редко |
| Превышение бюджета GigaChat | Средняя / Среднее | Мониторинг токенов через API метрики | Лимит контекста на диалог; ограничение длины истории (sliding window 10 сообщений) | Низкий для PoC-трафика |
| SQL-инъекция в user_memory | Низкая / Высокое | Code review; тесты | Параметризованные запросы (SQLite placeholder ?); никаких f-string в SQL | Очень низкий |
| Telegram webhook спуфинг | Низкая / Высокое | Логирование всех входящих webhook запросов | Валидация X-Telegram-Bot-Api-Secret-Token header; HTTPS only | Низкий |
| Злоупотребление поиском (DoS через search node) | Низкая / Среднее | Метрики числа запросов к DuckDuckGo | Rate limiting по user_id (не более 10 поисковых запросов в минуту) | Низкий |
| Галлюцинации LLM в расчётах | Высокая / Высокое | Eval-набор с автоматической проверкой | Ипотечный расчёт детерминированный (формула), LLM только форматирует; предупреждение пользователю проверить данные | Средний для нерасчётных ответов |

## Политика логов

### Что логируется

- Входящие сообщения пользователя (user_id + timestamp + message text)
- Результат роутинга (выбранный node)
- Вызовы внешних инструментов (cbr_tool, search_tool) с параметрами и статусом
- Ошибки и исключения (stack trace)
- Latency каждого шага графа
- HTTP-запросы к FastAPI (метод, путь, статус, время ответа)

### Что НЕ логируется

- Полные ответы GigaChat (только статус вызова и число токенов)
- Содержимое user_memory (персональные данные)
- Credentials и токены (только статус авторизации)

### Уровни логирования

```
INFO  - штатные операции (роутинг, вызов инструментов, ответы)
WARN  - нестандартные ситуации (fallback, медленный ответ)
ERROR - исключения, недоступность внешних сервисов
DEBUG - только в dev-окружении (полные промпты, raw API responses)
```

### Хранение

- Логи в `flat_agent/logs/` в rolling файлах (ротация по 10 MB, хранение 7 дней)
- В продакшне рекомендуется отправка в централизованную систему (ELK/Loki)
- Логи с персональными данными не выгружаются с сервера без явного запроса

## Работа с персональными данными

### Что хранится

В `user_memory` таблице SQLite хранится:
- `user_id` - идентификатор из Telegram (числовой ID, не имя)
- `key` - тип факта ("name", "budget", "district", "family_size")
- `value` - значение факта, извлечённое агентом из диалога
- `updated_at` - timestamp

### Политика

- Пользователь может запросить удаление своих данных командой `/forget` (в roadmap)
- Данные не передаются третьим лицам
- Данные используются только для персонализации ответов в рамках диалога
- В PoC нет обработки специальных категорий персональных данных (здоровье, финансовые счета)
- Telegram user_id не является именем - деперсонализация не требуется для PoC

### Соответствие 152-ФЗ

- PoC хранит минимально необходимые данные
- Нет обработки паспортных данных, СНИЛС, банковских реквизитов
- Пользователь добровольно сообщает факты в диалоге

## Защита от injection и подтверждение действий

### Prompt Injection

Архитектурные меры:
1. Разделение ролей в промпте: system instructions vs user data строго разделены
2. Результаты поиска и содержимое CSV передаются как `tool_result`, а не как `user_message`
3. LangGraph nodes получают только необходимый контекст, не полный стек сообщений
4. GigaChat не имеет инструментов для записи файлов или выполнения системных команд

Мониторинг:
- Логирование нестандартно длинных входящих сообщений (> 2000 символов)
- Alert при попытке использовать специальные символы промптов (`###`, `<|im_start|>` и т.п.)

### Подтверждение действий

В PoC все действия агента информационные (read-only по отношению к внешним системам):
- Агент читает данные с ЦБ РФ, но не отправляет туда ничего
- Агент ищет в DuckDuckGo, но не публикует
- Агент пишет только в user_memory и checkpoints (внутренняя БД)

**Принцип:** агент не совершает необратимых действий без явного запроса пользователя.

Для будущих версий (если появится интеграция с внешними сервисами):
- Двухшаговое подтверждение для любых write-операций
- Явное отображение планируемого действия перед выполнением
- Возможность отмены на любом шаге

### Rate Limiting

| Endpoint | Лимит |
|---|---|
| POST /api/v1/chat | 30 запросов/минуту на IP |
| Telegram webhook | 20 сообщений/минуту на user_id |
| DuckDuckGo поиск | 10 запросов/минуту на user_id |

Превышение лимита: HTTP 429 с `Retry-After` header.
